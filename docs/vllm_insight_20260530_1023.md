## 核心洞察

本期 vLLM 社区 30 天内产生了 30 条 RFC 和 17 条架构设计类 issue，占 noteworthy 总量的 72%。这个现象反映出 vLLM 正处于从"快速迭代的推理引擎"向"架构规范化平台"转变的关键阶段。三条主线尤为突出：**注意力/编译器架构重构**、**分布式推理架构深化**、**AMD ROCm 生态的全面博弈**。

**注意力与编译器架构**正经历自底向上的重组。#42449（Attention Backend 重构 RFC）和 #42082（KV Cache 布局标准化）试图从架构层面消除 attention backend 与 KV connector 间的紧耦合，这是 vLLM 历史上最大规模的注意力子系统重构。#43224（编译器融合手动化 RFC）则标志着 vLLM 对 torch.compile 策略的重新评估——主张将全图编译拆解为显式手动融合，逐个移植编译 pass 到模型代码中。配合 #42826（FlashAttention prefill/decode 分拆）、#41804（RMSNorm 阻止 Triton 融合）、#42545（Triton Tensor Descriptor 采纳），整个注意力+编译栈正在经历一场从编译器驱动到显式架构驱动的范式转换。

**分布式推理架构**同样密集演进。#42515（外部弹性 EP 扩展）和 #43470（PD 瓶颈感知自适应）分别从专家并行和 prefill/decode 分离两个维度推进弹性调度。#42109（分离式推测解码）、#43807（废弃 kv_both 强制 P/D 显式角色）、#42729（derender 端点分离 detokenization）都在推动 vLLM 的分布式架构向更细粒度的角色分离演进。而 #41893（Pod 快照 API）和 #43918（训练侧权重传输解耦）则从云基础设施和 RL 框架集成的角度扩展了 vLLM 的系统边界。

**AMD ROCm 生态**本期出现了令人担忧的态势。v0.21.0 虽宣称支持 DeepSeek V4 on ROCm（#42876），但官方镜像在 MI350X 上实际无法启动；Kimi K2.5/K2.6 性能倒退 38%（#43153）、MiniMax-M2.5 吞吐衰退 38%（#43029）、消费级 Blackwell 缺 CUTLASS MoE 后端（#43507）等问题接连暴露。这反映出 vLLM 在多厂商 GPU 支持上面临严峻的质量保障挑战——快速迭代的架构变更正在不断制造新的硬件兼容性断层。#43916（将 AMD 纳入 CI 回归体系）是缓解这一问题的关键举措。

此外，RL/推理融合是本期的新兴交叉领域。#41733（2026 Q2 RL Roadmap）和 #41821（RL rollout 自适应调度）表明 vLLM 正在将强化学习训练负载纳入一等支持对象，这将深刻影响调度器、内存管理和 KV 传输的架构设计。安全方面，#42216（V0 pickle CVE）和 #42591（/v2/embed API Key 绕过）提醒我们快速迭代中遗留代码和新增端点的安全审计不容忽视。
